# 待处理问题（⑤-⑬）

来源：逻辑 review + Codex review，已确认方向的 ①-④ 不在此列。

## ⑤ followup_round 字段没有赋值规则

**位置：** `schema.md` 第26行，`SKILL.md` P2  
**问题：** schema.md 里有 `followup_round`（追问层级，主问题为0），但 P2 切分规则完全没有说如何计算赋值。实现时只能自行补猜。  
**可选方案：** 在 P2 切分规则里补一行："主问题 followup_round = 0，每深一层 +1（Q→追问为1，追问→再追问为2）"；或直接从 schema 删掉这个字段（如果下游没消费它）。

---

## ⑥ question_summary 在普通题中用途不明

**位置：** `schema.md` batch_N_result.json，`SKILL.md` P3 第一步  
**问题：** SKILL.md 只在反向提问分支定义了 `question_summary`（候选人提问摘要），但 batch_N_result.json schema 对所有题目都包含此字段。普通题要不要填、填了用在哪里都没说。  
**可选方案：** 在 SKILL.md P3 第一步补一句"普通题 question_summary 留 null"，或从 batch_N_result.json schema 里把这个字段移到"仅反向提问时"的注释下。

## ⑦ strength/weakness 数量约束与模板不一致

**位置：** `output_template.md` 第40-46行，`SKILL.md` P3 第五步  
**问题：** output_template.md 写死了两条 strength、两条 weakness 作为示例，但 SKILL.md 没有数量约束，且允许表达质量问题为空时 weakness 留空。高分题或无问题题目无法按模板稳定渲染，容易出现凑条目的情况。  
**可选方案：** 把模板里的示例改为「一条或多条」的表达（如 `- [strength，可有多条]`），或在 SKILL.md 里补"strength ≥1 条，weakness 视实际情况可为空"。

## ⑧ P4a 标题与实际筛选逻辑不一致

**位置：** `SKILL.md` P4a，`output_template.md` 预测题章节  
**问题：** P4a 章节标题是「基于短板的追问预测」，但规则要求把 `has_followups = true` 的主问题也纳入，即使 score ≥ 4。这些高分题没有短板，answer_tip 也没有对应写作指引，章节标题会误导读者。  
**可选方案 A：** 改章节标题为「高关注度追问预测」或类似表述，覆盖"弱题"和"面试官重点深挖"两种情况。  
**可选方案 B：** 保留标题，在 P4a 规则里为 has_followups=true 高分题补一条 answer_tip 指引（如"面试官对此话题持续关注，建议准备更多细节或追问应对"）。

## ⑨ P1 confidence=medium 行为未定义

**位置：** `SKILL.md` P1  
**问题：** P1 明确定义了 high（直接继续）和 low（扩展到前20行）的处理方式，但 medium 完全没有说明，模型会自行决定。  
**可选方案：** 补一行"medium 直接继续，不扩展行数，但如有模糊说话人在报告开头注明置信度偏低"。

## ⑩ P4b 无公司名时的精确行为

**位置：** `SKILL.md` P4b  
**问题：** P4b 触发条件是"用户提供了岗位 JD"，但 WebSearch 搜索串是「[公司名] [岗位] 面经」，无公司名时 WebSearch 只能搜通用面经，结果质量大降。SKILL.md 只在 WebSearch 失败兜底里提到"公司名未提供"，没有在执行前做前置判断。  
**可选方案：** 在 P4b 开头加一条：若公司名未提供，跳过 WebSearch，只基于 JD 直接生成题目（不需要搜索依据），并在报告注明"题目基于 JD 推断，未参考面经"。

## ⑪ 报告日期来源未明确

**位置：** `output_template.md` 文件名和报告头部，`SKILL.md` P5

**可选方案：** 在 P5 补一句"日期取当前执行日期（YYYYMMDD）"，或在 P0 字段表加一个选填的"面试日期"字段。

## ⑫ Q 标题直接渲染完整 question 可能极长

**位置：** `output_template.md` 第27行，`SKILL.md` P2 切分规则第2条  
**问题：** P2 切分规则规定多子问题合并时 question 取"所有子问题拼接后的完整文本，用「；」分隔"。这个完整文本直接渲染到 `#### Q1：[question]` 标题里，遇到多连问时可能出现几百字的标题。  
**可选方案：** output_template.md 标题改用 `question_summary`（≤20字摘要），完整问题放到卡片正文第一个字段 `**提问：** [question]`。需要同时在 P3 第一步补充"普通题也要生成 question_summary"。

## ⑬ 两处规格文字自相矛盾

**位置：** `SKILL.md` P0 第44行 vs 第62行；P5 第266行

**矛盾一：** P0 字段表说文字稿"需含说话人标注"，但同一节检查2又允许无标注文字稿在用户确认后继续分析。前者是硬性要求，后者是软性兜底，两者语气不一致。  
**可选方案：** 把字段表里"需含说话人标注"改为"建议含说话人标注；如无标注，系统将尝试自动推断"。

**矛盾二：** P5 说"对话中只输出一行摘要"，紧接着又要求追加练习入口（多行文字）。  
**可选方案：** 改为"对话中只输出报告摘要和练习入口，不回显报告全文"。
