# 题型参考手册

本文件是 P3 逐题分析的题型依据。确定 primary_label 后，后续分析必须参考对应题型：理解考察意图、定位 strength/weakness/suggestion，并生成 optimized_answer。1-5 分的排他性映射与评分边界见 `scoring_rubric.md`。

---

## 使用规则

- **primary_label 决定主标准**：按主标签选择对应题型；secondary_label 只作为补充，不覆盖主标签。
- **intent 必须贴合题型**：先判断该题型通常考察什么，再结合岗位/JD 写本题意图。
- **点评必须贴合题型**：strength/weakness/suggestion 要围绕该题型的关键证据，不要只评价表达是否流畅。
- **optimized_answer 必须贴合题型框架**：按对应题型的应对框架重组原回答，不虚构经历和数据。
- **不要混用框架**：STAR 只适用于行为类/简历挖掘类；工作情景类重在拆解路径，业务类重在事实依据和逻辑链，个人类重在真实度与岗位匹配。

---

## 行为类

**定义：** 要求候选人回忆一段真实经历，以验证某项具体能力。核心特征是"举例"——面试官要的不是你的观点，而是你真实做过什么。本质上是宝洁八大问及其变体。

### 宝洁八大问

- 领导力：请举例说明你在一项团队活动中如何采取主动性，并且起到领导者的作用，最终获得你所希望的结果。
- 高目标：举例说明，你如何制定了一个很高的目标，并且最终实现了它。
- 划定关键点：请详细描述一个情景，在这个情景中你必须搜集相关信息，划定关键点，并且决定依照哪些步骤能够达到所期望的结果。
- 说服力：举例说明你是怎样用事实促使他人与你达成一致意见的。
- 团队合作：举例证明你可以和他人合作，共同实现一个重要目标。
- 问题决策：请举例，你是怎样评估一种情况，并将注意力集中在关键问题的解决。
- 创新力：举例证明，你的一个创意曾经对一个项目的成功起到至关重要的作用。
- 学习能力：举例说明你怎样获得一种技能，并将其转化为实践。


### 应对框架：STAR

行为类问题的黄金结构。评分时，重点看四个要素是否齐全、是否有深度：

- **Situation（背景）**：交代场景，简短即可，不是重点。失分常见问题：背景铺垫过长，占用了大量回答时间
- **Task（任务）**：你的任务是什么，同时你要知道WHY（为什么要做这个任务，可以从具体的上身到宏观的层面，体现你的思考）。失分常见问题：只说了团队任务，没说自己的任务。说了自己的任务，但不知道为什么团队要做这个任务。
- **Action（行动）**：你具体做了哪些事（是"你"而不是"我们"），描述结构化。失分常见问题：只说结论（"我提升了效率"）没说过程；行动描述模糊（"我积极推动了各方沟通"）；或描述没有层次感，过于冗长零碎。
- **Result（结果）**：量化结果，或至少定性说明影响范围。失分常见问题：没有结果，或只说"最终成功了"没有数据支撑。
- **反思**：此项为可选项，可以适当展示自己的反思与成长。

### 点评关注点

- Action 是核心：细节越充分，说服力越强。"我做了三件事：第一……第二……第三……"远优于"我积极推进"
- Result 是底线：缺失 Result 会显著影响说服力，具体评分边界见 `scoring_rubric.md`
- 个人贡献要清晰：面试官要评估的是你，不是你的团队。

### 优化回答要点

- 按 STAR 重组，不机械写标题，但顺序应包含背景、任务、个人行动、结果
- Action 部分用「我做了什么 + 为什么这样做 + 怎么推进」展开
- 原回答缺少数据时，用「（可补充：具体数字/时间）」占位，不编造结果
- 如果原回答已有反思，可放在结尾；没有反思时不要凭空添加重大感悟

## 工作情景类

**定义：** 给定一个假设场景，考察候选人的应对思路和决策框架。不要求回忆真实经历，重点是"你会怎么做"而非"你做过什么"。

### 典型题举例

| 场景类型 | 典型题干 |
|---|---|
| 数据异常 | 你负责的核心指标本周突然下滑 20%，你怎么分析？ |
| 跨部门协作冲突 | 你需要推动一个项目，但关键部门不配合，你怎么办？ |
| 资源约束 | 老板突然告诉你项目截止日期提前一个月，但人手和预算不变，你怎么处理？ |
| 优先级取舍 | 同时来了三个紧急需求，产品、运营、技术各说各的最重要，你怎么决策？ |
| 用户投诉/危机 | 产品刚上线出现大规模用户投诉，你作为负责人第一步做什么？ |
| 从零搭建 | 你被安排负责公司从没做过的新业务，没有现成团队和资源，怎么启动？ |

### 应对框架

好的工作情景类回答有两个特征：**路径清晰**（学会拆分问题的维度和阶段，比如按照时间维度、阶段（长短期）、内外等通用维度结合具体的其他维度，知道第一步做什么、第二步做什么）、**考虑约束**（现实条件下的权衡，尤其是要考虑成本约束）。

推荐结构：

1. **定义问题**：确认目标是什么，判断当前核心矛盾（不要急着给方案）
2. **拆解路径**：分层、分类或分阶段（如长短期）分析可能的原因/方向，说清楚优先看哪个、为什么
3. **提出方案**：针对最可能的原因或最关键的约束，给出具体行动
4. **说明权衡**：承认方案的局限性，或说明备选项
5. **验证效果**：说明如何观察结果、复盘调整或设置下一步指标

### 点评关注点

- **未拆解路径直接给结论** 是最常见的失分点，具体评分边界见 `scoring_rubric.md`。例：面试官问指标下滑怎么分析，候选人直接说"我会去找用户聊"——跳过了分析框架
- **未考虑约束条件**：只给理想解，不考虑成本、时间、人力、政治阻力，说明实战经验不足。比如面试官问：如果有人说我们的算法推荐不够准确，你会做什么？面试者回答：做用户调研和AB实验（第一步应该是成本最低的办法，比如先分析后台历史数据。）
- **结构化加分**：能说清楚"我会按 X 维度拆解，优先看 Y，因为……"的候选人明显高于泛泛而谈的候选人
- 和行为类不同，工作情景类**不强制要求 Result**，重点在思维过程

### 优化回答要点

- 不写成过去经历复盘，除非候选人原回答主动提到经历
- 先定义问题和目标，再分层拆解，不要直接给单点方案
- 必须体现现实约束，如时间、人力、数据可得性、跨部门成本或风险
- 结尾补充验证方式：看什么指标、如何判断方案有效、失败后如何调整

## 个人类

**定义：** 指向价值观、自我认知、职业规划，不要求回忆具体事件，考察候选人对自己的理解深度。

### 典型题举例

- 你最大的优点/缺点是什么？
- 五年后你希望自己在哪里？
- 你的职业规划是什么？
- 周围的人怎么评价你？
- 你为什么选择这个行业/岗位/公司？
- 你最看重工作中的什么？

### 应对思路

个人类问题没有统一框架，但有两个共同陷阱：

- **过于笼统**："我是一个有责任心的人，做事认真负责"——这类回答对任何岗位都成立，等于没说
- **过于完美**：缺点说成变相优点（"我太追求完美了"），面试官一眼识破

好的回答应当：**有具体案例支撑**（哪怕一个小细节）+ **与岗位相关**（把优势和应聘方向连接起来）+ **真实可信**（不需要精心包装，直接说反而有说服力）

### 点评关注点

- 个人类不强制 STAR，但如果候选人能用一个简短例子支撑观点，比纯粹陈述有力得多
- 职业规划类问题：说得太远太模糊（"我想成为行业领导者"）和说得太短视（"就想先把这份工作做好"）都是弱回答；好的回答能体现对自身成长路径的真实思考
- 与岗位的连接度：如果候选人的规划和应聘岗位方向明显不符，应在 weakness 中指出

### 优化回答要点

- 回答应真实、具体、克制，避免过度包装或模板化口号
- 用一个简短例子或具体偏好支撑自我判断
- 明确连接目标岗位：为什么这个特质、动机或规划和岗位相关
- 缺点类问题要给出真实限制和改进动作，不要把优点包装成缺点

## 简历挖掘类

**定义：** 从候选人简历中的具体条目出发追问，考察经历的真实性和深度。

### 典型特征

题干通常包含：「你简历上写的……」「这个项目你具体负责什么」「这段经历……能展开说说吗」

### 应对思路

本质上是行为类的一个子集，同样用 STAR 结构。

### 点评关注点

- 重点验证经历真实性：候选人是否能说清项目背景、约束、关键决策和细节
- 责任边界必须清楚：候选人具体负责什么，团队其他人负责什么
- 项目理解要有深度：不能只背简历描述，要能解释为什么这样做、指标如何变化、复盘得到什么
- 追问下仍能自洽：前后信息不能明显矛盾，不能用「不是我做的」「不太清楚」回避核心问题

### 优化回答要点

- 先明确自己在项目中的角色和责任边界
- 按「背景/目标 → 我的职责 → 关键行动 → 结果/复盘」展开
- 对非本人负责的部分，可以说明上下游理解，而不是直接回避
- 原回答缺少项目数据时，用「（可补充：具体数字/时间）」占位

## 业务类

**定义：** 围绕目标岗位、行业、公司、产品或业务问题的认知性问题。核心不是展示经历，而是展示判断依据和岗位理解。

### 典型题举例

- 你怎么看这个行业的发展趋势？
- 你认为我们产品目前最大的问题是什么？
- 如果让你负责这个业务，你会优先做什么？
- 你怎么理解这个岗位的核心价值？

### 应对思路

好的业务类回答应当有清晰观点，但不能只有观点。推荐结构：

1. **先给判断**：明确说出核心观点或优先级
2. **说明依据**：用事实、数据、用户行为、竞品观察或岗位逻辑支撑观点
3. **展开影响**：说明这个判断对目标岗位、业务增长、用户体验或组织协作意味着什么
4. **承认边界**：在信息不足时说明假设条件，不把猜测包装成事实

### 点评关注点

- 观点必须有依据：不能只说「我觉得」「我认为」而没有事实或逻辑链
- 逻辑链要完整：从现象到原因，再到影响或行动建议
- 岗位视角要明确：回答应体现候选人理解目标岗位如何创造价值
- 深度加分来自取舍：能说明优先级、成本收益、风险或反例，比单纯罗列观点更有说服力

### 优化回答要点

- 用「观点 → 依据 → 影响/建议 → 边界」重组回答
- 原回答缺少数据或来源时，用「（可补充：数据来源/观察依据）」占位
- 不虚构行业数据、公司内部信息或竞品事实
- 如果题目要求方案，至少说明一个优先级和选择理由

## 其他

**定义：** 无法稳定归入上述题型的问题，或题目意图较弱、偏开放、偏流程说明但仍需要候选人回应的问题。

### 应对思路

- 先判断面试官真正想确认什么信息
- 回答要直接、具体、完整，不要绕开问题
- 如果题目本身信息不足，可以先说明假设，再给回答

### 点评关注点

- 是否正面回答了问题
- 是否提供了足够信息供面试官判断
- 是否与目标岗位或当前面试上下文相关

### 优化回答要点

- 保留候选人原意，压缩无关铺垫
- 补齐必要背景、判断依据或结论
- 不强行套用 STAR、业务分析框架或职业规划框架
