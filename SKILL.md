# 面试镜 · Interview Mirror

将带说话人标注的面试文字稿，转化为一份结构化 Markdown 复盘报告。

---

## 首次运行引导

如果用户触发了本技能但未提供文字稿，先输出以下引导消息，然后等待补充：

> **面试镜已就绪**
>
> 请提供以下内容开始分析：
> 1. **面试文字稿**（必填）：粘贴文本，或提供 .txt / .md / .docx 文件路径
> 2. **应聘岗位**（必填）：如「产品经理」「数据分析师」
> 3. 可选：公司名称、面试轮次、简历摘要、岗位 JD（提供 JD 后将额外预测高频面试题）
>
> 提供上述信息后，我将自动运行完整的五阶段分析并生成复盘报告，无需逐步确认。

如果用户已提供文字稿和岗位，跳过此步骤，直接进入 P0 预检查。

---

## 输入说明

开始前，确认以下信息（未提供的可询问用户，允许用户不提供岗位JD）：

| 字段 | 必填 | 说明 |
|---|---|---|
| 文字稿 | 必填 | 文件路径或粘贴的文本；需含说话人标注（如 `S1`、`说话人1` 等） |
| 应聘岗位 | 必填 | 如「产品经理」「数据分析师」 |
| 公司名称 | 选填 | 如「字节跳动」 |
| 面试轮次 | 选填 | 如「一面」「HR 面」 |
| 简历背景 | 选填 | 候选人简历摘要，100字以内 |
| 岗位 JD | 选填 | 岗位职责描述；提供后才执行 P4b（高频题预测） |

读取文字稿后，按以下步骤顺序执行：P0 预检查 → P1 识别说话人 → P2 切分问答单元 → P3 逐题分析 → P4a/P4b 预测追问 → P5 生成整体报告，最后写入 Markdown 文件。

---

## P0：文字稿预检查

在正式进入 P1 前，对文字稿做两项快速检查：

**检查 1：是否为面试对话**
判断文字稿是否为面试场景（问答结构、职业相关内容、有明确的提问-回答交替）。若内容看起来不像面试对话（如产品文档、新闻报道、会议纪要等），询问用户：

> 「上传的文本看起来不像面试文字稿。您是否上传了错误的文件？如确实是面试记录，请确认后我继续分析。」

等待用户确认后继续，或等待重新上传。

**检查 2：是否含说话人标注**
扫描全文，检测是否存在任何形式的说话人标注（如 `S1:`、`说话人1:`、`A:`、`Interviewer:` 等）。若完全不含标注，告知用户：

> 「文字稿中未检测到说话人标注，这可能影响问答单元切分的准确性。是否尝试基于句式特征（提问/陈述模式）自动推断说话人角色后继续？」

- 用户确认继续 → 进入 P1，confidence 默认设为 low，报告开头注明「文字稿无说话人标注，角色识别基于句式推断，请人工核查」
- 用户取消 → 停止分析，建议用户补充说话人标注后重新提交

---

## P1：识别说话人角色

取文字稿前 8 行对话，判断哪个说话人标签是面试官、哪个是候选人。

**判断依据：**
- 面试官通常先开口，进行开场或提出第一个问题
- 面试官句式多为提问、引导：「来介绍一下」「说说你的看法」「为什么」
- 候选人句式多为陈述、描述经历

**多面试官处理：**
若识别到超过两个说话人，判断哪些是面试官角色（提问方）、哪个是候选人（回答方）。将所有面试官标签统一归并为 `interviewer`，并在报告开头注明「识别到多名面试官，已合并处理」。

**输出：**
```
interviewer: S1（或文字稿中实际使用的标签；多人时列出所有归并标签）
candidate: S2
confidence: high / medium / low
```

如果置信度为 low（片段不足4句、开场以候选人自我介绍为主、或存在多名说话人），扩展到前20行重新判断。若仍无法确定，在报告开头注明「说话人识别置信度低，以下分析基于推断，请人工核查」，并继续执行 P2（P2 的切分逻辑同样依赖此映射，低置信度时可能切分不准）。

---

## P2：切分问答单元

以面试官每次提问为起点、候选人完整回答为终点，切分为若干「提问—回答」单元。

**切分规则：**
1. 以面试官的每次提问为一个单元的起点，候选人完整回答为终点
2. 若面试官连续提出多个问题后候选人统一回答，合并为一个单元，question 取**所有子问题拼接后的完整文本**（保留各子问题，用「；」分隔），确保 P3 分析时能看到完整考察意图
3. 追问链：面试官在候选人回答后就同一话题继续深问，`is_followup = true`，`parent_unit_id` 指向主问题；主问题同时标记 `has_followups = true`
4. 候选人向面试官的反问环节：`is_candidate_question = true`，question 填候选人的提问原文，answer 置为 null，interviewer_response_summary 填面试官回答摘要

**忽略（不生成单元）：**
- 开场寒暄与结束语
- 流程性说明（「下面我来介绍一下今天的面试流程」）

**内部数据结构（供后续步骤使用）：**
```
单元列表：[
  {
    unit_id,             // 题目编号，从1开始连续
    is_followup,         // 是否为追问题
    parent_unit_id,      // 追问题指向的主问题 unit_id，否则为 null
    has_followups,       // 该题是否被追问过（P4a 依据此字段）
    followup_round,      // 追问层级，主问题为 0
    is_candidate_question,
    question,            // 完整提问文本（含所有子问题）；反向提问时为候选人提问原文
    answer,              // 候选人完整回答原文；反向提问时为 null
    interviewer_response_summary,  // 仅反向提问（is_candidate_question = true）时使用：面试官回答摘要
    // P3 分析后补充以下字段：
    primary_label, secondary_label, intent,
    answer_summary, score,
    strength, weakness, suggestion, optimized_answer,
    question_summary
  }
]
```

## P3：逐题分析

对每个问答单元逐一执行以下六步分析。**按顺序执行，步骤间有联动规则。**

### 第一步：判断是否为反向提问
如果 `is_candidate_question = true`，只输出简化分析：
- primary_label: 反向提问
- score: null（不评分）
- answer_summary: 面试官回答摘要，1句话（取自 interviewer_response_summary）
- strength/weakness/suggestion/optimized_answer: 均为空
- question_summary: 候选人提问摘要，不超过30字

### 第二步：打标签
从以下标签中选 1-2 个（最多1主1次，只有1个明确标签时不补次标签）：

| 标签 | 定义 | 典型题 |
|---|---|---|
| 个人类 | 指向价值观、自我认知、职业规划，不要求回忆具体事件 | 个人特质、职业规划、周围人评价 |
| 行为类 | 要求回忆真实经历以验证某种能力，通常含「举例」「描述一次」结构 | 领导力经历、高目标达成、团队协作 |
| 工作情景类 | 给定假设场景，考察应对思路 | 指标下滑如何分析、跨部门冲突 |
| 业务类 | 围绕目标岗位、行业或具体产品的知识性/认知性问题 | 行业格局、行业发展 |
| 简历挖掘类 | 从简历具体条目出发追问 | 「你简历上写的……」「这个项目你负责什么」 |
| 反向提问 | 候选人向面试官提问 | — |
| 其他 | 兜底分类 | — |

**排他规则：**
- 要求回忆真实经历 → 行为类；否则 → 工作情景类
- 从简历条目出发 → 简历挖掘类；泛泛要求举例 → 行为类
- 问题核心指向候选人自身价值观/认知/规划时，优先打个人类

### 第三步：分析面试官意图
结合问题内容、岗位名称、JD（若提供）分析底层考察目的，不超过50字。

### 第四步：生成回答摘要
提炼候选人回答核心内容，2-3句，保留关键结论和数据。

### 第五步：评分与点评

**联动规则（必须遵守）：**
- intent（第三步）决定本题核心评价标准，weakness 必须围绕该标准展开
- weakness 必须是 score 偏低的主要原因；score 越低，weakness 描述的问题越严重
- suggestion 必须直接修复 weakness[0]，使用动作句式（如「用 STAR 补充……」）
- 如果本题是追问，weakness/suggestion 必须判断是否补齐了上一题的缺口

**语气要求：**
- weakness 使用建设性语气，避免「你不应该……」等否定式表达
- suggestion 给出具体可操作的改进方向
  - 反例：「建议抓住题目重点核心，避免泛泛而谈。」
  - 正例：「面试官问题是：与他人存在分歧如何克服，重点是说清楚分歧是什么、你做了哪些行动、有什么结果。你的回答一直在谈团队宏观性难题，没有深入到自己的具体例子。」

**strength 写作规则：**
- 必须引用候选人回答中的具体内容或数据
- 禁止「表达清晰」「逻辑好」「回答完整」等空泛表述
- 每条不超过30字

**评分标准（1-5分）：**

| 分数 | 标准 |
|---|---|
| 1分 | 严重跑题、极度简略（如只回答「是/否/不知道」），或暴露严重逻辑/价值观问题 |
| 2分 | 回答了表面问题但缺乏结构，没有具体例子/数据，逻辑明显断层，或有推脱责任的表述 |
| 3分 | 达到基本要求。有基本逻辑（STAR雏形），有定性经验描述，但缺乏深度、未量化结果 |
| 4分 | 结构严谨，有清晰动作细节和可衡量的结果，有效论证了专业能力 |
| 5分 | 在4分基础上展现高阶思维：全局观、深度复盘反思、方法论沉淀、对业务的深刻洞察 |

**扣分上限（硬性规则，优先于描述性判断）：**
- 无任何具体例子或证据支撑 → 最高 3 分
- 语气词过多（额、那个、就是……）→ 最高 3 分
- 行为类/简历挖掘类：Result 缺失 → 最高 2 分；个人贡献不清晰 → 最高 3 分；Action 缺乏细节 → 最高 3 分
- 工作情景类：未拆解路径直接给结论，或未考虑任何约束条件 → 最高 3 分
- 业务类：观点全无逻辑链或事实依据 → 最高 3 分
- 任何题型：出现明显推责表述（「不是我做的」「那时候不归我管」）→ 最高 1 分

### 第六步：生成优化参考回答
基于候选人原回答重组优化，必须针对 weakness 中的问题进行改进。若原回答缺少数据，用「（可补充：具体数字/时间）」占位，不得虚构数字或经历。
- 行为类/简历挖掘类：不超过 300 字（STAR 结构完整展开需要更多空间）
- 其余题型：不超过 200 字

## P4a：基于短板的追问预测

对以下两类题目预测面试官下一轮最可能的追问方向：
- **score ≤ 3 分**的题目（表现薄弱，下轮可能被深挖）
- **has_followups = true** 的主问题（本轮已被追问，说明面试官对此话题高度关注）

注意：选取的是**主问题**（`has_followups = true`），而非追问题本身（`is_followup = true`）。

**弱追问题处理：**
若 `is_followup = true` 且 `score ≤ 3`，判断该追问暴露的是独立于主问题的新短板，还是主问题短板的延续：
- 新独立短板 → 单独生成一道预测题
- 主问题短板的延续 → 并入主问题的预测条目，不单独计数

**预测规则：**
1. 优先基于该题具体的 weakness 内容生成追问方向，不要泛泛预测
2. `has_followups = true` 的主问题优先纳入预测，即使 score ≥ 4 分
3. 每道弱题最多生成 1 道预测题，总数控制在 1-5 道
4. 弱题不足以支撑 3 道预测时，输出实际可支撑的数量，不要凑数

**answer_tip 要求：** 针对原有 weakness 给出具体改进方向；如涉及 STAR/数据量化/框架缺失等结构性问题，直接点明应补充的要素。

## P4b：基于 JD 的高频题预测（可选）

**仅当用户提供了岗位 JD 时执行。** 先使用 WebSearch 工具搜索「[公司名] [岗位] 面经 面试题」，然后基于搜索结果和 JD 提炼高频面试题。

**WebSearch 失败处理：**
若 WebSearch 工具不可用、搜索结果为空、或公司名未提供，跳过 P4b，在报告「预测题」章节注明「高频题预测不可用（搜索失败或信息不足）」，不影响其他步骤的输出。

**处理规则：**
1. 搜索结果是唯一题目证据来源；不得引入搜索结果中完全没出现的新考点
2. 公司、岗位、JD 是强过滤条件：即使某题出现频率较高，只要与目标岗位相关性弱，也必须过滤
3. 多份面经中出现相同或相似的题目，合并为一道，记录出现次数
4. 每个问题必须是完整问句，不允许关键词列表或不完整短语
5. 去除纯闲聊、薪资谈判等非考察性问题
6. 输出 3-5 道；合格题不足 3 道时，返回实际数量

## P5：生成整体报告

基于所有问答单元的分析结果，生成整体复盘报告。

### 一句话总结
格式固定：「[核心优势]，[关键制约因素]」，不超过50字，结合目标岗位特性，不要泛泛而谈。

示例（按岗位类型）：
- 产品类：「展现了出色的B端产品架构能力，但在商业化变现的思考深度上是当前主要制约。」
- 运营类：「活动策划思路清晰且有数据意识，但跨部门协作案例中个人贡献边界模糊，削弱了说服力。」
- 技术类：「后端工程基础扎实、问题定位能力突出，但系统设计题中缺乏全局视角。」

### 四维度评价
对以下四个维度进行整体文字评价（各2-3句话），**评价必须引用具体题目编号作为证据**（如「在 Q3 拆解指标时……」），并给出独立的 1-5 分（保留1位小数，不得简单平均各题分数）：

**1. 结构化与逻辑：** 回答的组织能力，是否善用框架（STAR/总分总），在复杂追问下是否保持逻辑连贯。

**2. 经验与量化结果：** 历史经验的含金量，是否用具体数据和案例支撑了个人核心贡献，而非停留在"参与者"视角。

**3. 业务/专业认知：** 对目标岗位的理解深度，行业洞察，专业方法论沉淀。**仅当存在「业务类」或「工作情景类」题目时输出；否则设为 null，在报告中渲染为 `-（本次未涉及）`。**

**4. 沟通与自我呈现：** 沟通气场、求职意愿的真实度、遇到不会的问题时的抗压反应。

### 核心竞争力（优势）
从所有题目中归纳，输出 2-4 条跨题目的共性优势。

### 致命扣分项（短板）
归纳影响面试通过率的 2-4 条核心短板。优先暴露「能力缺失」「业务认知不足」或「反复出现的表达缺陷」。每条注明关联题目编号，并说明对申请岗位的负面影响。

### 下一步行动（Action Items）
基于核心短板输出 2-4 条可执行的行动项，每条包含：
- **问题**：要解决的具体问题
- **行动**：提供具体练习框架或思路（如「采用 PREP 结构重写该案例，突出3个关键数据指标」）；拒绝「建议多加练习」这种空话
- **关联题目**：如 Q2、Q5

## 输出格式

所有分析完成后，写入文件 `面试复盘_[岗位]_[日期].md`（如 `面试复盘_产品经理_20260422.md`）。

报告头部（公司名、面试轮次若未提供则渲染为"未提供"）：

```markdown
# 面试复盘报告
[公司名或"未提供"] · [岗位名]
[面试轮次或"未提供"] · [日期]
综合评分 X.X / 5
```

总览部分：

```markdown
---

## 总览

### 一句话总结
[overall_summary]

### 维度评分

| 维度 | 评分 | 简评 |
| --- | --- | --- |
| 结构化与逻辑 | X.X/5 | [dim_structure 摘要] |
| 经验与量化结果 | X.X/5 | [dim_quantification 摘要] |
| 业务/专业认知 | X.X/5 或 - | [dim_business 摘要，若未涉及填"本次未涉及"] |
| 沟通与自我呈现 | X.X/5 | [dim_self_presentation 摘要] |

### 核心竞争力
- [优势1]
- [优势2]

### 致命扣分项
- [短板1]（Q2、Q5）：[对岗位的负面影响]
- [短板2]（Q3）：[对岗位的负面影响]
```

逐题复盘部分——每道题根据类型使用对应模板：

**普通问答题模板：**

```markdown
#### Q1：[question_summary]（[primary_label]）
```

## 执行注意事项

- **不要在每步骤完成后询问用户是否继续**，直接执行完整流水线，最后汇报结果
- P3 逐题分析时，追问题（is_followup = true）需在分析中注明「本题是对 Q[parent_unit_id] 的追问」，并判断候选人是否补齐了上一题的缺口

- 综合评分 = 所有有效题目（score 非 null）的平均分，保留1位小数；若无任何有效评分题（全为反向提问或切分失败），综合评分显示「-」并在报告开头注明「本次有效评分题目不足，综合评分不适用」
